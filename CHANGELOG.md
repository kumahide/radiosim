## [2.4] — 未リリース（ベータ）

テーマ「A-0（アンテナ初期指向 AZ/EL）＋出力層 v2（A4 ドロップイン）」。スクリーニング結果を **そのまま報告書へ綴じ込める A4 成果物** にし、経路に **初期指向 AZ/EL** を載せる一巡。レポートを HTML＋print CSS で「無加工で他の A4 成果物へ綴じ込める確定 1 枚」に作り込み、PDF 化はブラウザの Ctrl+P に委任（PDF エンジンは非導入）。

### 追加
- **レポート v2 ＝ A4 ドロップイン**: per-path（`report.html`）／summary（`summary.html`）を **A4 縦の印刷確定枠**（`@page A4`・`@media print`）で生成。画面では A4 用紙（`.sheet`）を描いて WYSIWYG に、印刷では Ctrl+P で **ゼロ依存 PDF 化**（PDF 生成エンジンを内製・依存追加しない＝哲学③⑤）。各ページに **自己同定ヘッダ/フッタ**（タイトル・案件名・生成日時・ID・版）を付す。印刷ダイアログでは「ヘッダーとフッターをオフ」を前提とする（ブラウザ挿入の印刷ヘッダは CSS 抑制不可のため自前ヘッダで代替）。
  - **per-path を A4 縦 1 枚に収める**: 本文を JS の「縮小フィット」で実測し、印字域を超える分だけ一様縮小して 1 頁に固定（重複・低価値行の削除、地図アスペクト 15:5、地形断面のデータ表を削除）。
- **案件情報（案件名・自由メモ）**: レポートの自己同定ヘッダに載る任意メタ情報を **ランチャーに入力欄を新設**（source of truth）。**シングル（保存時）もバッチも同じ値を踏襲**する。バッチでは Common Settings と同じく 🔒 読み取り専用スナップショットとして表示し、「↻ ランチャーから更新」で取り込む。案件名＝per-path/summary 両ヘッダ、自由メモ＝summary のヘッダ直下（単一レポートにも表示）。タイトル文字列そのものの自由編集は行わない（テンプレエディタ化しない・案件名フィールドで代替）。
- **アンテナ初期指向 AZ/EL（A-0）**: per-path レポートの Site Info に、相手局へ正対するための **真方位 AZ ＋仰角 EL** を両端ぶん表示。既存データ（座標・アンテナ高・地形標高・地球曲率）から幾何計算する純関数（`models.bearing_deg` / `models.elevation_angle_deg`）で、新規入力はゼロ。両端で別値（AZ は大圏逆方位、EL は高低差項の符号反転）。「初期指向。最終調整は現場の RSSI 最大化で」と注記し、あわせて **AZ が真方位である旨と磁気コンパス使用時の偏角補正（日本＝西偏約 7〜9°）** も注記する。
- **全経路を 1 枚に俯瞰する地図（summary）**: `report_map.render_paths_map`（north-up・回転なし）で、全端点の外接矩形を出力アスペクトへフィットさせた俯瞰地図を `summary.html` の 1 ページ目に埋め込む。経路は判定ステータスで色分け（OK 緑 / NG 橙赤 / ERROR 濃赤・台帳の行色と単一ソース `map_graphics.STATUS_RGB`）、中点に path_id ラベルを付す（本数が多いときはラベルを省く）。タイル取得に失敗したら地図を省いて注記（ベストエフォート）。

### 変更
- `APP_VERSION` を `"2.4b3"` に。
- **進捗トランスポートを 1 部品へ統合（2.4b3）**: 単一実行とバッチが各々で持っていた「キュー＋`after` ポーリング」を **`views/progress.py` の `ProgressPump`** に括り出した。**狙いは重複の削減ではなくライフサイクルの統一**＝従来バッチのポーラは生成時に 1 回起動して永久に回り、単一のポーラは実行ごとに起動・停止するという非対称があり、これ自体が B-006 の停止バグを生んだ。両者とも「実行のあいだだけ回す」に揃え、同種のバグが構造的に成立しないようにした。2 者の差は 2 つのパラメータに閉じている（単一＝最新 1 件だけ描画／バッチ＝`done`・`complete` が結果を運ぶので全件を順序どおり処理）。あわせて**実行中にウィンドウを閉じたとき破棄済みウィジェットへ `after` し続ける経路**も塞いだ。なお単一とバッチのパイプライン本体は統合しない（出口が対話グラフと確定成果物で異なる＝2.3 で確定した設計思想）。
- **summary 台帳の可読性を刷新**: `table-layout:auto` で各列を内容幅に合わせ（ヘッダを語中で折らない・長い数値がはみ出さない）、ヘッダの単位は 2 行目へ、列を縦罫線で分離、フッタを用紙下端へ。極端値を含む台帳でも A4 印字域に収める。
- **`path_id`・備考に文字数上限**: 長すぎる値は台帳の列幅を押し広げ A4 レイアウトを崩すため、実行前の `validate_rows` で弾く（手入力・CSV 取込の共通チョークポイント）。上限は path_id 16 文字・備考 40 文字。
- **`infrastructure.py` を分割**: `config.py`（アプリ設定）＋ `dem.py`（DEM/タイル取得・キャッシュ）へ分離（DEM 系を触る変更のためのモジュール整理）。

### 修正
- **単一実行の進捗表示が実態とずれていた（I-008・2.4b3）**: 標高取得が速くなった結果（B-006）、バーが担当する区間は暖機時 0.035 秒になった一方、その後の matplotlib 遅延 import とグラフ構築（計 約 0.6 秒・メインスレッドを占有）は**バーの管轄外**で、しかもその直前にラベルを「準備完了」へ戻していた。実際には固まっている間ずっと「準備完了」と表示していたことになる。グラフ描画に入る前に **「グラフを準備中…」** を表示し、グラフが画面に出る直前に待機状態へ戻すようにした。対話グラフは pyplot/TkAgg でメインスレッド必須のためワーカーへは出せず、偽の進捗も出さない（バッチの段階ラベルと同じ方針）。
- **ランチャーの「マップウィンドウ」ボタンが表示されない（B-007・2.4b3）**: ウィンドウ高さが `450x900` のリテラル固定だったため、2.4 で「案件情報」グループ（案件名・メモ）を足して必要高さが 931px になった時点で最下段のボタンが可視領域外へはみ出していた。ウィンドウは固定サイズなのでユーザーは広げて回避することもできない。高さのリテラルを廃し、**中身の必要量（`winfo_reqheight`）から決める**方式へ変更（下限 900px・上限は画面高さの 92%）。以後、入力グループを足しても追随する。
- **バッチのプログレスバーが実態を表さず、途中で GUI ごと固まる（B-006・2.4b2）**: 独立した 3 つの欠陥が重なっていた。①バーの駆動源が DEM 標高取得のみで、体感時間の大半（レポート描画）がバーの管轄外だった → 配分を「取得 15% / 描画 85%」とし、描画中は偽の進捗を出さず段階ラベルで示す。②成果物生成（per-path PNG/HTML/KML・サマリ地図/HTML/KML）を GUI コールバックで呼んでおり、その間メインスレッドが停止していた → `batch` のワーカースレッドへ移動（「matplotlib はメインスレッド必須」は `save_profile_png` が pyplot から `FigureCanvasAgg` へ移行した時点で**失効していた制約**）。③単一実行が進捗コールバックに `root.after` を直接渡しており、`fetch_elevations` がグローバルロック保持中に 1 サンプルごと呼ぶため Tcl 呼び出しが全ワーカーを直列化していた（同一条件で単一 ≒1.0 秒／バッチ ≒0.035 秒）→ バッチと同じキュー＋50ms ポーリング方式へ統一。
- **単一実行の進捗ラベルが完了後も残る（B-006 の修正中に作り込んだ回帰・2.4b2）**: 進捗をキュー化した際、停止フラグを下ろしても `after(50)` 済みのポーリングが 1 回残り、積み残しの進捗が完了表示を上書きしていた。キュー破棄と `_poll_progress` の早期 return の二重防御で解消（ワーカーは完了通知の後にも進捗を push しうるため、キュー破棄だけでは不足）。

- **バッチが地形キャッシュを素通りしていた（2.4b3）**: 単一実行は `fetch_elevations_cached`（TX/RX 座標＋サンプル数が同じなら DEM を再取得しない）を使うのに、バッチだけが素の `fetch_elevations` を呼んでおり、同一バッチを再実行するたびに DEM を取り直していた。バッチも単一と同じキャッシュ付き取得へ統一（1 行だけ直しての再実行でも、変えていない行の取得が消える）。実行前のタイル一括取得（プリフェッチ相）はバッチには設けない＝遅延取得でも総取得量は変わらず、得られるのは進捗の見え方だけのため。

## [2.3] — 2026-07-03

正式リリース。テーマ「座標入力の完成」。バッチのランチャー凍結（Phase D1）と地図からの連続追加
（Phase D2・地図をランチャー所有の単一インスタンスに一本化）で、座標入力体験を完成。
b1〜b2 のローカル検証と RC1〜RC2 のバイナリ実機確認（ローカルβ検証で見つかったバッチUI不具合を含む）
を経て正式版とした。

### 追加
- **バッチ「ランチャー凍結」方式（Phase D1）**: バッチの設計モデルを「シングル＝条件を詰める場／バッチ＝確定条件で成果物を作る場」に転換。ランチャー（シングル）が唯一の真実の源となり、バッチの各行は **行追加時点のランチャー欄を文字列コピーして凍結した確定リンク** になる。
  - **+ 行を追加** は、その時点のランチャー欄（座標・周波数・利得・アンテナ高）を凍結した行を生成する。
  - **共通設定を読み取り専用化**: ランチャーのスナップショットとして表示し、**「↻ ランチャーから更新」** ボタンで現在値を取り込む。
  - **per-row 往復（右クリックメニュー）**: 行ごとに **「→ シングルへ送る」**（座標＋RF をランチャーへロード）／**「⟳ RF をランチャーで更新」**（座標は保持し RF のみ書き戻し）／複製／削除。
  - **アンテナ利得の per-row 化**: `gain_tx` / `gain_rx` を経路ごとに指定可能なリンク識別属性として追加（バッチ表に2列・CSV の省略可列・`summary.csv` / `summary.html` に利得列）。未指定時は共通設定を継承。
- ランチャーに `load_batch_row`（→シングルへ送る の受け口）を追加し、`BatchBuilderWindow` に `config_provider` / `load_params` を注入。
- **連続追加モード（Phase D2）**: マップウィンドウに **「バッチへ連続追加」** モードを追加。地図で TX→RX のペアを置くたびにバッチ行が自動追加され、自動的に次の TX 待ちへリセットされる（+行 追加は不要）。RF（周波数・利得・アンテナ高）は **追加時点のランチャー値で凍結**。バッチ表にある全パスは地図上に TX/RX マーカー＋経路線＋水平距離で表示する。
  - **地図はアプリ唯一のインスタンス（ランチャー所有）に一本化**。「バッチからも地図を開く」第2インスタンスは作らない（ランチャーが本筋・バッチは従属する受け皿）。連続追加モードに入るとランチャーがバッチウィンドウを開く/前面化し、append 先として配線する。バッチを閉じると地図は座標入力モードへ戻る。
  - `MapWindow` のピック宛先を 2 種類の「座標シンク」で表現: 単一書き戻し（ランチャー＝start/end 欄・構築時固定）と連続 append（バッチ＝`append_path`/`existing_paths`・モード移行時に取得）。
  - **バッチの行変更を地図へリアルタイム反映**: 削除・全クリア・CSVインポート・追加・複製、および座標セルの編集確定（FocusOut/Enter）で地図の確定パス表示が即座に追従する（バッチ→ランチャー→地図のコールバック）。並べ替えはパス集合が不変なので再描画しない。一括再構築（インポート/並べ替え）は逐次通知を抑止して終了後に1回だけ更新。
  - **確定パスの端点表示を TX=塗りドット／RX=方位矢じりに変更**（`map_graphics.arrow_icon`）。従来の TX/RX 文字ラベル＋塗り/白抜きノードは、TX/RX が近接・同一座標だと文字もノードも重なって判別不能だった。送受を別形状（ドット↔矢じり）で表し、矢じりは TX→RX 方位を指すので向きも分かる。文字ラベルはアクティブ選択中のパスのみ残す。
  - マウスホイールのスクロール波及を修正（`bind_all` がアプリ全体に効くため、別ウィンドウのホイール操作でバッチ表が一緒にスクロールしていた問題。ポインタが表上のときだけスクロールするよう限定）。
